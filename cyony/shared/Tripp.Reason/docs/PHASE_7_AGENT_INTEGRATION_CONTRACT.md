# Phase 7 OpenClaw + Hermes Integration Contract

> Locked before any external-agent code exists. Defines how Tripp.Reason coordinates with OpenClaw Tripp, Hermes Cyony, and OpenClaw Echo as bounded external workers while preserving orchestration, approval, reporting, and safety boundaries.

## Purpose

Phase 7 connects external agents to Tripp.Reason as bounded workers. Tripp.Reason does not replace these agents — it coordinates, routes, records, gates, and audits their work. Each external agent receives structured task packets, returns structured result packets, and never bypasses Tripp.Reason's safety infrastructure.

**Operating model:**

```
Operator (Eddie) = Final approver
Tripp.Reason      = Runtime / Orchestrator / Tool boundary / Report brain
OpenClaw Tripp    = Lead/Supervisor worker (cloud, disciplined logic)
Hermes Cyony      = Creative sandbox worker (cloud, exploration/prototyping)
OpenClaw Echo     = Local Warden/Auditor (local-machine trust boundary)
```

## Non-Goals

- ❌ No direct unrestricted repo access for external agents
- ❌ No Hermes production write access by default
- ❌ No bypass of ApprovalGate — all mutation goes through Tripp.Reason tools
- ❌ No bypass of Warden review — every external result is reviewed
- ❌ No hidden autonomous background execution
- ❌ No external agent spawning other agents (no recursive spawning)
- ❌ No recursive swarm spawning (workers cannot spawn workers)
- ❌ No secrets passed to external agents (redaction required)
- ❌ No remote control of local machine without explicit boundary
- ❌ No replacement of Tripp.Reason runtime
- ❌ No provider sprawl — uses existing provider config
- ❌ No new provider implementations required by external agents
- ❌ No direct swarm imports from external agent packages

## Agent Roles

### OpenClaw Tripp — Controller / Supervisor / Lead

**Nature:** Cloud agent. Disciplined logic, reasoning, control role.

**Allowed:**
- Planning and task decomposition
- Supervising build packets (reviewing, routing)
- Code review of proposed changes
- Bounded implementation if explicitly approved in task packet
- Returning structured result packets with explicit proposed changes
- Recommending next steps

**Forbidden:**
- Bypassing Tripp.Reason orchestration
- Direct uncontrolled repo mutation (all writes through Tripp.Reason tools)
- Spawning agents outside Tripp.Reason's control
- Overriding operator approval decisions
- Accessing secrets, API keys, or credentials
- Final merge authority (operator retains)

### Hermes Cyony — Creative / Builder / Swarmhandler

**Nature:** Cloud sandbox agent. Agentic, exploratory, creative.

**Allowed:**
- Exploration and investigation of codebases, patterns, architectures
- Generating architecture alternatives and design proposals
- Prototyping in sandbox environment (no production writes)
- Creative worker packet generation (proposals, ideas, workflows)
- Prompt and workflow experimentation
- Returning structured proposal packets with findings, risks, recommendations

**Forbidden:**
- Production repo write access by default (sandbox-only writes)
- Final approval authority (all output is proposal, not authorization)
- Secret access, API keys, credentials
- Destructive operations (delete, overwrite without backup, system modification)
- Direct merge authority
- Final architecture approval without Warden/operator review
- Self-promoting sandbox output to production

### OpenClaw Echo — Warden / Auditor / Trace

**Nature:** Local agent. Runs on local machine. Trust boundary: can read local state.

**Allowed:**
- Reviewing reports generated by Tripp.Reason and external agents
- Checking scope drift — did the result match the original objective?
- Doctrine compliance checking — does output comply with DOCTRINE.md?
- Safety trace — can trace every mutation back to an approved decision
- Local audit of repo state, logs, and reports
- Recommending pass/fail/block on external agent output
- Flagging violations for operator attention

**Forbidden:**
- Overriding operator approval — Eddie is final approver
- Silently mutating repo state (all mutations through Tripp.Reason tools)
- Bypassing Tripp.Reason logs and audit trail
- Replacing the Warden component in swarm pipeline (augments, not replaces)
- Accessing external network or cloud resources

## Trust Zones

### Zone Definitions

| Zone | Location | Trust Level | Access |
|------|----------|-------------|--------|
| **Local Runtime Zone** | `packages/*` on local machine | Full trust | Tools, filesystem, DB, reports |
| **Cloud Sandbox Zone** | Hermes remote environment | Minimal trust | Copied context only, no local access |
| **Cloud Worker Zone** | OpenClaw Tripp remote environment | Bounded trust | Task packets + context bundles only |
| **Local Warden Zone** | Echo on local machine | Audit trust | Read repo state, logs, reports; no mutation |
| **Production Repo Zone** | Local filesystem under workdir | Operator-guarded | Mutations only through ApprovalGate |
| **Report/Archive Zone** | `reports/` directory | Read-only for agents | External agents can read their own reports |

### Zone Rules

1. **Hermes cloud sandbox** receives copied context and task packets — NOT broad local filesystem access. Context is minimized and secrets are redacted before transmission.
2. **Echo warden** can audit local reports and repo state but still works through declared boundaries — no arbitrary filesystem traversal outside the workdir.
3. **OpenClaw Tripp** receives bounded packets (task + context bundle) and returns structured result packets. No streaming access, no persistent connection.
4. **Production repo mutation** must happen through Tripp.Reason tools and ApprovalGate — external agents never write directly.
5. **Report/archive zone** is read-only for external agents after report generation. They cannot modify historical reports.

## External Worker Packet Contract

### ExternalAgentTaskPacket

Extends the existing `TaskPacket` from `@tripp-reason/swarm` with agent-specific fields:

```typescript
interface ExternalAgentTaskPacket {
  // ── Existing TaskPacket fields (from swarm/types.ts) ──
  id: string;
  role: WorkerRole;
  title: string;
  objective: string;
  scope: string;
  allowedFiles?: string[];
  forbiddenFiles?: string[];
  allowedTools?: string[];
  forbiddenTools?: string[];
  modelTier: ModelTierLabel;
  riskLevel: RiskLevel;
  timeoutMs: number;
  requiresApproval: boolean;
  contextRefs?: string[];
  expectedOutput: string;
  dependsOn?: string[];

  // ── External agent extensions ──
  agentId: string;           // Which agent (tripp | cyony | echo)
  agentRole: string;         // Agent's role label
  agentProvider: string;     // openclaw | hermes | openclaw-local
  contextBundle: string;     // Redacted, minimized context for cloud agents
  outputFormat: string;      // Expected output format (json | markdown | both)
  reportRequired: boolean;   // Whether agent must produce a report
  returnChannel: string;     // inbox | http | mcp | manual
}
```

### ExternalAgentResultPacket

Extends the existing `ResultPacket` from `@tripp-reason/swarm`:

```typescript
interface ExternalAgentResultPacket {
  // ── Core identification ──
  taskId: string;
  agentId: string;
  agentRole: string;

  // ── Status ──
  status: "pass" | "partial" | "fail";

  // ── Output ──
  summary: string;              // Human-readable summary
  findings: Finding[];          // Structured findings
  proposedChanges: ProposedChange[];  // Explicit, reviewable diffs
  filesTouched: string[];       // Files agent claims to have affected
  risks: RiskNote[];            // Agent's own risk assessment

  // ── Validation ──
  validation: string;           // Self-validation explanation
  reportPath?: string;          // Path to generated report (if reportRequired)
  artifacts?: unknown;          // Optional raw output (capped, sanitized)

  // ── Review flags ──
  needsHumanReview: boolean;    // Agent explicitly requests operator review
  recommendedNextStep: string;  // What agent recommends as next action
}
```

### Contract Requirements

1. **Structured output required** — no vague prose as the only result. Every result packet must contain structured fields.
2. **All proposed mutations must be explicit** — `proposedChanges` array with file, diff, reason. No "I changed some files."
3. **All uncertainty must be marked** — `needsHumanReview: true` when agent is uncertain. Risk notes for every risk.
4. **Timeout enforced** — `timeoutMs` from task packet is the hard cap. Late responses are discarded.
5. **Fail-closed** — if any required field is missing or malformed, the result is treated as FAIL.

## Communication / Transport Options

### Option Comparison

| Transport | Pros | Cons | Startup Complexity |
|-----------|------|------|---------------------|
| **File dropbox / shared folder** | Simple, auditable, no network config | Polling latency, file locking | Lowest |
| **HTTP local endpoint** | Real-time, standard, observable | Needs server on both sides | Medium |
| **MCP-style bridge** | Tool-native, structured, tested pattern | Protocol overhead, connection management | Medium-High |
| **Shared agent bus** | Pub/sub, multi-agent, scalable | Complex, over-engineered for 3 agents | High |
| **Manual copy-paste fallback** | Zero infrastructure | Human bottleneck, error-prone | Lowest (fallback only) |

### Recommended Starting Transport

**Phase 7 recommendation: File/message packet exchange with inbox/outbox folders.**

Rationale:
- No always-on remote control required
- Explicit inbox/outbox is auditable and debuggable
- Works with any agent that can read/write files
- No network configuration needed for first integration
- Aligns with Tripp.Reason's file-system-native design
- HTTP and MCP can be added later without changing the packet contract

### Proposed Folder Structure

```
.tripp/agents/
├── inbox/           # Tasks sent to external agents
│   ├── tripp/       # Tasks for OpenClaw Tripp
│   ├── cyony/       # Tasks for Hermes Cyony
│   └── echo/        # Tasks for OpenClaw Echo
├── outbox/          # Results returned by external agents
│   ├── tripp/
│   ├── cyony/
│   └── echo/
├── reports/         # Reports generated by agents
├── archive/         # Completed/processed packets (audit trail)
└── context/         # Redacted context bundles for cloud agents
```

Packet naming: `{agentId}_{taskId}_{timestamp}.json`

## Approval Boundary

### Rules

1. **External agents cannot approve their own mutations.** All proposed changes must pass through Tripp.Reason's ApprovalGate.
2. **Hermes cannot self-promote sandbox output to production.** Sandbox output is always treated as proposal, never as authorization.
3. **Echo/Warden can recommend pass/fail/block but Eddie remains final approver.** Warden is advisory.
4. **Tripp.Reason ApprovalGate remains authoritative for all tool execution.** External agents have no direct tool access — they return proposed changes.
5. **If approval system unavailable, fail closed.** No external agent output is applied without operator review.
6. **Every approval decision is logged and auditable.** Who approved what, when, and why.

### Approval Flow Diagram

```
ExternalAgentResultPacket arrives
  ↓
Warden (Echo) reviews → recommends PASS / FAIL / BLOCK
  ↓
Tripp.Reason merges result into swarm pipeline
  ↓
Merger consolidates with other results
  ↓
Warden (swarm) performs final safety review
  ↓
Operator (Eddie) reviews and approves final output
  ↓
ApprovalGate executes approved mutations
  ↓
Report generated with full audit trail
```

## Sandbox Rules for Hermes (Cyony)

Hermes operates in a cloud sandbox with strict boundaries:

### Allowed
- Exploration and analysis of provided context
- Generating architecture proposals, design documents, workflow ideas
- Creative prototyping (within sandbox — no production writes)
- Producing structured proposal packets with findings and risks
- Experimenting with prompts and configurations

### Forbidden
- Sandbox-only writes — no production repo mutation
- Output treated as proposal/report only — not authorization
- Destructive operations forbidden (delete, overwrite, system modification)
- No secret access — API keys, credentials, tokens must be redacted from context
- No direct merge authority — merge requires Warden + operator review
- No final architecture authority without Warden/operator sign-off
- No network access beyond what sandbox provides

### Context Rules for Hermes

Before sending context to Hermes cloud sandbox:
1. Redact all secrets (API keys, tokens, passwords, private URLs)
2. Minimize context to what the task requires (no broad repo dumps)
3. Include only files listed in `allowedFiles` — never `forbiddenFiles`
4. Strip credentials from configuration files
5. Log what context was sent (for audit trail)

## Warden Rules (Echo)

Echo operates locally and reviews external agent output:

### Review Checklist
1. Scope drift — does output match the original objective?
2. Doctrine compliance — does output comply with DOCTRINE.md?
3. Safety — are there destructive or risky proposed changes?
4. Completeness — are required fields present and valid?
5. Consistency — does output conflict with other results?
6. Auditability — can every proposed change be traced to its source?

### Warden Actions
- **PASS:** Output is clean, safe, complete → recommend proceed
- **PARTIAL:** Minor issues found → flag for operator attention, recommend proceed with caution
- **FAIL:** Critical issues found → recommend block, require operator override to proceed

### Warden Constraints
- Cannot override operator approval — Eddie is final authority
- Cannot silently mutate repo — all changes through Tripp.Reason tools
- Cannot bypass Tripp.Reason logs — every action is recorded
- Cannot access external network or cloud resources

## Swarm Integration

External agents become swarm worker backends in Phase 7F:

### Architecture

```
Orchestrator
  ├── Fake workers (deterministic, local)
  ├── ReasonLoop workers (real, local)
  └── External agent workers (remote, bounded)  ← NEW in Phase 7F
```

### Integration Rules

1. **External agents are worker backends** — the orchestrator selects the backend type per task.
2. **Task packet remains the same** — the existing `TaskPacket` from `@tripp-reason/swarm` is the contract.
3. **Result packet remains the same** — the existing `ResultPacket` from `@tripp-reason/swarm` is the contract.
4. **External agents must satisfy the same ResultPacket contract** — no special-casing.
5. **Orchestrator chooses** local fake → real ReasonLoop → external agent backend based on task properties.
6. **External agents cannot spawn subagents** unless Tripp.Reason explicitly delegates (Phase 7+ only, with hard limits).
7. **No recursive spawning** — an external agent working on a task cannot request its own external agent.

### Package Boundary

```
swarm ← core, shared           (existing, unchanged)
external-agents ← shared       (NEW: imports shared for packet types)
external-agents ← swarm        (type-only: imports TaskPacket, ResultPacket types)
core ↛ external-agents         (NEVER — core is external-agent-free)
swarm → external-agents        (swarm MAY import external-agents for backend selection)
```

The external agent package (`packages/external-agents/`) imports shared for contracts and swarm for type compatibility. It does NOT import core directly — ReasonLoop is not needed for inbox/outbox packet exchange.

## Security Requirements

### Packet Security
- **No secrets in packets** — all credentials, tokens, API keys must be redacted before transmission
- **Context minimization** — send only what the task requires, not broad repo dumps
- **Redaction before cloud agents** — secrets are stripped at Tripp.Reason boundary, not at the agent
- **Path boundaries** — file paths must be within allowed scope; traversal attacks blocked

### Runtime Security
- **Report retention** — all external agent output is archived for audit
- **Audit logs** — every packet sent/received is logged with timestamps and content hashes
- **No unbounded file attachments** — file sizes capped; malicious payloads rejected
- **No arbitrary command execution** — agents return proposals, not executable commands
- **No hidden background work** — every task is tracked, observable, and auditable
- **Timeout required** — every external task has a hard timeout; late responses discarded

### Trust Model
- **Fail closed** — if any security check fails, the operation is blocked
- **Operator approval remains final** — no automated approval chain can override Eddie
- **External agents are always untrusted** — even Tripp (cloud) is treated as untrusted input
- **Echo (local) has audit trust but not mutation trust** — can read, cannot write
