# Phase 7B Agent Bus Protocol

> File-based local packet bus for bounded external-agent coordination.

## 1. Purpose

Phase 7B establishes the foundational folder protocol for exchanging task and result packets between Tripp.Reason and external agents (OpenClaw Tripp, Hermes Cyony, OpenClaw Echo). This is a **local, file-based, human-auditable** transport that requires no always-on network connections, no background daemons, and no runtime coupling.

The protocol defines **where** packets go, **what** they may contain, **how** they are named, and **what rules** govern their lifecycle. It does NOT implement packet creation, reading, validation, or agent adapters — those come in later phases.

## 2. Scope

Phase 7B scope is strictly limited to:

- Creating the folder structure under `.tripp/agents/`
- Writing README documentation for each folder
- Defining the packet lifecycle, naming conventions, and content rules
- Documenting agent role routing and approval boundaries
- Establishing security and failure handling rules

## 3. Non-Goals

Phase 7B explicitly does NOT include:

- ❌ Live agent integration or connectors
- ❌ Adapter implementation (OpenClaw, Hermes, Echo)
- ❌ HTTP agent bridge or cloud transport
- ❌ MCP bridge changes
- ❌ Server routes for agent interaction
- ❌ Dashboard panel for agent status
- ❌ CLI commands for packet management
- ❌ Repo mutation by external agents
- ❌ ApprovalGate bypass or modification
- ❌ Background watchers, daemons, or file monitors
- ❌ Any runtime behavior change
- ❌ Any new npm dependencies

## 4. Folder Layout

```
.tripp/agents/
├── README.md        ← Root documentation
├── inbox/           ← Task packets waiting for external agents
├── outbox/          ← Result packets returned by external agents
├── reports/         ← Agent-generated or Tripp.Reason-generated review reports
├── archive/         ← Completed accepted packet/result cycles
└── rejected/        ← Failed, blocked, unsafe, malformed, or rejected packets
```

### inbox/

- **Purpose:** Holds task packets ready for an external agent to pick up.
- **Allowed:** JSON task packets with objective, scope, agent role, constraints, and expected output format.
- **Prohibited:** Secrets, raw `.env` files, broad repo dumps, unscoped paths, executables, mutation-authorizing files.
- **Lifecycle:** Task created → placed in inbox → external agent reads → result written to outbox → task moved to archive (accepted) or rejected (blocked).

### outbox/

- **Purpose:** Holds result packets returned by external agents after completing a task.
- **Allowed:** JSON result packets with task ID, agent ID, status, summary, findings, proposed changes, risks, validation.
- **Prohibited:** Executable code, direct mutation commands, secrets, self-authorizing approval claims.
- **Lifecycle:** Agent writes result → reviewed by Echo/Warden → accepted or rejected → moved to archive or rejected.

### reports/

- **Purpose:** Holds human-readable reports generated by agents or Tripp.Reason's Warden.
- **Allowed:** Markdown reports, JSON review summaries. Echo/Warden review reports.
- **Prohibited:** Secrets, raw repo content (use references), executable code, approval claims.
- **Lifecycle:** Generated after review → read by operator → may be archived alongside accepted packets.

### archive/

- **Purpose:** Stores completed accepted packet/result cycles for audit and traceability.
- **Allowed:** Accepted task packets, result packets, and associated review reports.
- **Prohibited:** Pending packets, rejected packets, secrets.
- **Lifecycle:** Accepted items moved here → read-only → retained until explicit cleanup policy is designed.

### rejected/

- **Purpose:** Stores failed, blocked, unsafe, malformed, or rejected packets for audit and learning.
- **Allowed:** Malformed packets, unsafe packets, out-of-scope packets, failed results, operator-rejected results, Warden-blocked results, timeout results.
- **Prohibited:** Accepted or pending packets, secrets (should never have been in the bus).
- **Lifecycle:** Rejected items moved here → read-only → retained for audit → may be reviewed for learning.

## 5. Packet Lifecycle

The intended full lifecycle (future phases):

```
1. TASK CREATION
   Operator or Tripp.Reason creates a task packet (JSON) with explicit scope, constraints, and agent assignment.

2. TASK DISPATCH
   Task packet is placed in inbox/ for the target agent to discover.

3. AGENT EXECUTION
   External agent reads the task packet from inbox/. Agent performs bounded work within declared scope.

4. RESULT DELIVERY
   External agent writes a result packet (JSON) to outbox/. Result includes status, findings, proposed changes, risks.

5. REVIEW
   Echo/Warden reviews the result packet. Review report is written to reports/.
   Human operator (Eddie) may also review.

6. DECISION
   Operator approves, rejects, or requests revision.
   - Approved → move task + result + review to archive/
   - Rejected → move to rejected/ with rejection reason

7. MUTATION (if applicable)
   Any proposed changes that were approved must still pass through Tripp.Reason's ApprovalGate.
   External agents never mutate the repo directly.

8. CLOSURE
   Cycle is complete. All artifacts are in archive/ or rejected/. Audit trail is preserved.
```

**Key principle:** Every step is observable. Every packet is auditable. Nothing happens silently.

## 6. Naming Convention

Recommended naming pattern for future implementation:

| Packet Type | Pattern | Example |
|-------------|---------|---------|
| Task | `task-YYYYMMDD-HHMMSS-{agentRole}-{short-slug}.json` | `task-20260603-143022-tripp-review-auth.json` |
| Result | `result-YYYYMMDD-HHMMSS-{agentRole}-{short-slug}.json` | `result-20260603-144500-tripp-review-auth.json` |
| Review | `review-YYYYMMDD-HHMMSS-echo-{short-slug}.md` | `review-20260603-150000-echo-auth-audit.md` |
| Report | `report-YYYYMMDD-HHMMSS-{agentRole}-{short-slug}.md` | `report-20260603-151000-cyony-auth-proposal.md` |

Rules:
- Filenames should be deterministic enough to audit
- No spaces — use hyphens
- Use lowercase slugs
- No secrets in filenames
- Include agent role for routing clarity
- Timestamps enable chronological sorting

## 7. Packet Content Rules

### Task Packets (JSON)

Required fields (future schema):
- `id` — unique task identifier
- `agentId` — target agent (tripp | cyony | echo)
- `agentRole` — agent role label
- `title` — short human-readable title
- `objective` — detailed description of the task
- `scope` — explicit scope boundaries
- `allowedFiles` / `forbiddenFiles` — path constraints
- `timeoutMs` — hard deadline
- `riskLevel` — low | medium | high
- `expectedOutput` — what the result should contain

### Result Packets (JSON)

Required fields (future schema):
- `taskId` — reference to the original task
- `agentId` — which agent produced this result
- `status` — pass | partial | fail
- `summary` — human-readable summary
- `findings` — structured findings array
- `proposedChanges` — explicit, reviewable diffs (file, diff, reason)
- `risks` — agent's own risk assessment
- `validation` — self-validation explanation
- `needsHumanReview` — boolean flag for uncertainty

### Content Rules

- **JSON for packets** — machine-parseable, structured
- **Markdown for reports** — human-readable
- **No secrets** — no API keys, tokens, passwords, credentials
- **No raw .env files** — strip all secrets before placement
- **No broad repo dumps** — scoped context only
- **No credentials or private tokens**
- **No unscoped file content** — only referenced paths
- **Include scope, objective, constraints, allowed paths, denied paths, and expected output** in every task packet

## 8. Agent Role Routing

Default routing by agent role:

### OpenClaw Tripp (supervisor/lead)
- **Receives:** Planning, supervision, code review, bounded implementation packets
- **Returns:** Structured review results, implementation proposals, planning documents
- **Authority:** Can recommend — cannot approve mutations

### Hermes Cyony (creative builder)
- **Receives:** Creative, prototype, exploration, architecture proposal packets
- **Returns:** Design proposals, alternatives, findings, creative output
- **Authority:** Sandbox-only — no production write authority. Output is always proposal.

### OpenClaw Echo (warden/auditor)
- **Receives:** Review, audit, drift-check, doctrine-compliance packets
- **Returns:** Warden reports, violation flags, pass/fail/block recommendations
- **Authority:** Advisory only — cannot override operator. Cannot mutate repo.

### Unknown / Unmatched Roles
- **Fail closed.** Packets addressed to unknown agent roles are rejected.

## 9. Approval Boundary

The Agent Bus is a transport layer — it does NOT grant approval authority.

- **Packets are not approvals.** A task packet does not authorize its own completion.
- **Reports are not approvals.** A result report documents what happened, not what should happen.
- **External result files are not executable authority.** No file in `outbox/` grants mutation rights.
- **ApprovalGate remains authoritative.** All proposed changes must pass through Tripp.Reason's ApprovalGate before any repo mutation occurs.
- **Eddie remains final approver.** No automated chain, no file presence, no agent output overrides the operator.
- **Every approval is logged and auditable.** Who approved what, when, and why.

**If the approval system is unavailable, fail closed.** No mutation occurs without explicit operator approval.

## 10. Security Rules

| Rule | Enforcement |
|------|-------------|
| No secrets in Agent Bus | Operator responsibility; future tooling will validate |
| No direct shell authority through packets | Packets describe work, not commands |
| No cloud agent local filesystem access | Cloud agents receive only redacted context bundles |
| No mutation authority implied by outbox result | All mutations through ApprovalGate |
| No recursive agent spawning | Agents cannot request new agent tasks |
| No hidden autonomous background work | Every task is tracked and auditable |
| Malformed packets fail closed | Rejected with reason |
| Unknown agent roles fail closed | Rejected |
| Packets requesting secrets are rejected | Moved to `rejected/` |
| Packets requesting direct mutation are rejected or converted to proposal-only | Operator decides |

## 11. Failure Handling

| Failure Mode | Action |
|-------------|--------|
| Malformed packet (invalid JSON, missing fields) | Move to `rejected/` with parse error reason |
| Unsafe packet (requests secrets, direct mutation) | Move to `rejected/` with safety violation reason |
| Out-of-scope packet (exceeds agent authority) | Move to `rejected/` with scope violation reason |
| Unknown agent role | Move to `rejected/` with unknown-role reason |
| Agent timeout (no response within deadline) | Mark as failed, move original to `rejected/` |
| Agent returns status "fail" | Move to `rejected/` with agent's failure reason |
| Operator rejects result | Move to `rejected/` with operator's rejection reason |
| Warden blocks result | Move to `rejected/` with warden's violation details |

All rejections must preserve traceability:
- Original packet is preserved
- Reason for rejection is documented
- Timestamps are preserved
- Rejection is auditable
